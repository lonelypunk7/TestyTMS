# Правила создания тест-кейса в TestY

## Смысл кейсов из Smoke (suite 341)

Suite «Smoke» — критические end-to-end сценарии авторизации Wi-Fi. Все кейсы проверяют **один бизнес-процесс**:

> Пользователь подключается к Wi-Fi → открывается captive portal → проходит авторизацию → получает доступ в интернет.

### 4 способа авторизации (4 кейса):
1. **ЕОС** — единая учётная запись
2. **MosID** — mos.ru
3. **YandexID** — Яндекс
4. **Госуслуги** — ЕСИА

### Flow тестирования:
```
Подключение к SSID → Captive Portal → Выбор способа → Авторизация → Доступ в интернет
```

## Структура тест-кейса

### Поле `name` — название
- **Конкретное**, описывает **что проверяется**
- Без нумерации (номер = ID в TestY)
- Не более 1 строки

**Плохо:** `Проверка`
**Хорошо:** `Автооткрытие captive portal после подключения к SSID`
**Хорошо:** `Успешная авторизация через MosID`

### Поле `description` — контекст
- Краткое описание **цели** теста
- Формат: `Цель: проверить, что ...`

**Пример:**
```
Цель: проверить, что после подключения к целевой Wi-Fi сети открывается страница авторизации.
```

### Поле `setup` — предусловия
- Описываем **состояние системы ДО теста**, а не действия
- Каждый пункт — с новой строки
- Начинаем с `Предусловия:`

**Пример:**
```
Предусловия:
Точка доступа активна.
Устройство не авторизовано в этой сети (чистая сession).
На устройстве включен WI-FI
```

**Другой пример:**
```
Предусловия:
Устройство подключено к SSID.
Авторизация не выполнена.
```

### Поле `scenario` — шаги (опционально, при is_steps: true)
- Краткое резюме, не дублирует steps
- Используется для простых кейсов

### Поле `expected` — ожидаемый результат (опционально, при is_steps: true)
- Краткое резюме, не дублирует steps

### Поле `is_steps` — **всегда `true`**
- Каждый кейс имеет массив `steps`
- Каждый шаг — отдельный объект

### Поле `steps` — детальные шаги
Каждый шаг содержит:
- `name` — краткое название шага
- `scenario` — что делает пользователь
- `expected` — ожидаемый результат шага
- `sort_order` — порядок (1, 2, 3...)

**Правила:**
- Каждый шаг — **одно действие**
- Не более 5 шагов на кейс
- `name` — без нумерации
- `scenario` — конкретные действия
- `expected` — один проверяемый факт

### Поле `labels` — метки
- Обычно пустой массив `[]`
- При необходимости — существующие лейблы проекта

## Типовые сценарии Smoke-кейсов

### 1. Проверка открытия портала (кейс 4557, 4560)
```
setup: Точка доступа активна, устройство не авторизовано, WI-FI включён
steps:
  1. Подключиться к SSID
     → scenario: пользователь выбирает SSID и подключается
     → expected: подключено к Wi-Fi
  2. Подождать captive portal
     → scenario: не выполняет действий, ожидает
     → expected: открывается captive portal
```

### 2. Проверка авторизации способом (кейс 4563-4566)
```
setup: Валидные данные для способа авторизации, устройство не авторизовано
steps:
  1. Выбрать способ и завершить авторизацию
     → scenario: пользователь выбирает способ, выполняет шаги, отправляет форму
     → expected: авторизация успешна, доступ выдан
```

### 3. Проверка ограничения доступа до авторизации (кейс 4567)
```
setup: Устройство подключено к SSID, авторизация не выполнена
steps:
  1. Открыть внешний сайт без авторизации
     → scenario: пользователь открывает внешний ресурс без логина
     → expected: доступ ограничен, удерживается в flow авторизации
```

### 4. Проверка доступа после авторизации (кейс 4568)
```
setup: Устройство подключено, авторизация выполнена
steps:
  1. Открыть внешние сайты
     → scenario: пользователь открывает HTTPS-ресурсы
     → expected: открываются без редиректа на портал
```

## Общие правила

### ✅ Делать
- `is_steps: true` **всегда**
- Каждый шаг — **одно действие**
- Конкретные названия (не "что-то", а "captive portal", "MosID", "Wi-Fi")
- `setup` — состояние системы, не действия
- `description` — начинается с `Цель: проверить, что ...`
- Не более 5 шагов на кейс
- Каждый step.expected — **один проверяемый факт**

### ❌ Не делать
- `is_steps: false` — всегда `true`
- Более 5 шагов на кейс
- Смешивать действия и ожидания
- Дублировать setup в steps
- Оставлять пустым step.expected
- Писать шаги в одну строку

### ⚠️ Важно: не дублировать шаги из Setup
- **`setup` описывает состояние ДО теста** — например: «Устройство подключено к SSID», «Открыт captive portal»
- **В `steps` пишем только действия ПОСЛЕ setup**
- Если в setup указано «Устройство подключено к SSID», то в steps не нужно писать шаг «Подключиться к SSID»
- Если в setup указано «Captive portal открылся», то в steps не нужно писать шаг «Открыть captive portal»
- Каждый шаг в steps — это **действие, которое пользователь выполняет в рамках текущего теста**, а не предварительное условие
